# CLAUDE.md — Course Development Workflow

## 🎯 Purpose

Define consistent and automatable rules for generating course materials from a central syllabus.
This file provides Claude with precise instructions for managing course unit files and generation order.

## 🎓 重要说明：课程设计的专业流程

**课程创建和管理工作应遵循专业教学设计原则：**

- 课程设计需要考虑教学目标的递进性和知识体系的完整性
- 从课程大纲创建到章节文件生成，应确保教学内容的一致性和连贯性
- 重视实践环节的设计，确保理论与应用的有机结合
- 根据具体需求选择合适的方法和工具进行课程开发

--- 

## 📁 目录结构规则

- 在项目根目录创建 `课程大纲.md`。
- 按照课程大纲的每一个章节在根目录下创建文件夹，文件夹以“章节序号-章节名称”命名，例如：`01-基础概念`、`02-数据结构`。
- 每个章节目录路径格式：`<ROOT>/<章节序号-章节名称>/`。
- 每个章节下必须包含以下核心文件：
  1. `教案.md` —— 用于该章节的教学设计（目标、重点、课堂任务）
  2. `教学手册.md` —— 教师按照教案执行课堂活动的操作手册（教学脚本、时间分配、质量检查点）
  3. `学生手册.md` —— 学生完成课堂任务的指导手册（学习目标 → 操作步骤 → 提交物）
- 所有路径应使用相对引用，并在章节目录名中保留“章节序号-章节名称”的标识。
- 对用户需求生成的其它文件，根据需要创建文件夹进行组织。
---

## ⚙️ 生成顺序（严格）

生成或更新章节文件时必须遵循以下顺序：

`教案.md → 教学手册.md → 学生手册.md`

可选（在四个核心文件完成后）：

- 根据需要生成其它文件，并在章节目录或专用子目录中进行组织

---

## 🔄 工作流（课程大纲驱动）

### Step 0 — 课程大纲创建与管理

**当用户提出创建新课程需求时，必须通过用户交互明确关键参数后，进行系统性的课程大纲创建和设计：**

#### 0.1 需求确认与参数设定
**必须通过用户交互工具（如AskUserQuestion）确认以下关键信息：**

1. **课时安排确认**
   - 总课时数：____课时
   - 每课时时长：____分钟（通常为45、50、60或90分钟）
   - 每周课时安排：____课时/周
   - 课程总时长：____周

2. **学习目标层级设定**
   - **知识目标**：学生应掌握的理论知识
   - **技能目标**：学生应具备的实操能力
   - **应用目标**：学生应能完成的项目或任务
   - **进阶目标**：学有余力学生的拓展方向

3. **课程深度与广度平衡**
   - 理论深度：基础概念/原理深入/前沿技术
   - 实践比重：理论实践比（如3:7, 4:6, 5:5）
   - 项目复杂度：入门级/中级/高级/企业级

4. **评估方式偏好**
   - 平时作业占比：____%
   - 期中项目占比：____%
   - 期末项目占比：____%
   - 其他评估方式：____

**交互设计原则：**
- 提供合理的默认选项供用户选择
- 对关键参数进行解释说明
- 允许用户自定义输入特殊需求
- 确保收集的信息足够支撑后续的课程设计

#### 0.2 课程结构设计
- 基于确认的参数设计合理的课程结构和知识体系
- 按照学习递进原则安排章节顺序
- 确保每个课时的学习目标明确可测量
- 平衡理论教学与实践操作的比例

**课时设计指导原则：**
- **45分钟课时**：适合单一技能点教学，理论+简单练习
- **50分钟课时**：标准课时，可包含1-2个知识点的深度讲解
- **60分钟课时**：适合复杂概念教学，理论+实践+总结
- **90分钟课时**：适合项目驱动教学，可完成完整的小项目

**学习目标设定要求：**
- 每课时学习目标不超过3个主要目标
- 目标描述应使用明确的动词（掌握、能够、完成、设计等）
- 理论目标与实践目标相结合
- 确保目标可测量、可验证

#### 0.3 大纲创建与验证
- 创建完整的 `课程大纲.md`，包含：课程基本信息、目标、详细章节安排、评估方式、教学资源等
- 确保课时分配合理，每课时目标清晰
- 与用户确认大纲是否符合预期，根据反馈调整

- 可选参数：`章节根目录`（默认为项目根目录）。

### Step 1 — 大纲校验

- 比较 `课程大纲.md` 的修改时间（`mtime`）与各章节的 `last_synced` 记录。
- 标记任何缺失或过期的章节为"待创建/更新"。

### Step 2 — 解析大纲

- 提取章节序号与名称（如：`01-基础概念`、`02-数据结构`）。
- 构建目标章节列表（新增 + 修改）。

### Step 3 — 目录设置

对每个目标章节：

- 若目录不存在 → 创建 `<章节序号-章节名称>/`。
- 确保 3 个核心文件存在（空白或从模板初始化）：
  - `教案.md`、`教学手册.md`、`学生手册.md`
- 所有路径使用相对引用，并保持章节目录名中的“章节序号-章节名称”标识。

### Step 4 — 生成核心文件

**按严格顺序生成或更新：**

`教案.md → 教学手册.md → 学生手册.md `

需要生成：

1. **教案.md**：包含六个部分
   学习目标 / 核心概念 / 课堂活动 / 练习任务 / 作业 / 评估标准
2. **教学手册.md**：基于教案的分钟级教学指导（教学脚本、时间分配、质量检查点）
3. **学生手册.md**：可执行的学生指引（学习目标 → 步骤 → 提交物）

可选（核心文件完成后）：

- 可根据需要生成其它补充文件，并在章节目录或专用子目录中进行组织

### Step 5 — 对齐记录

**需要确保：**

每个核心文件应包含 YAML 头：

```yaml
version: v1.0
last_synced: YYYY-MM-DD
source_syllabus: "课程大纲.md@<mtime 或 commit>"
```

- 检查各文件的标题、教学目标与产出是否与课程大纲一致
- 确保教学内容的连贯性和递进性
- 验证学习目标的可测量性和可达性

## ✅ 质量与安全注意事项

- 未经明确允许，不覆盖人工编写内容。
- 写入前确认必需的元数据（YAML 头）存在。
- 通过版本控制确保可复现性。
- 在处理不相关任务前清理上下文（如使用 `/clear`）。
